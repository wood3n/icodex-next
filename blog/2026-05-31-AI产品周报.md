本期 AI 产品周报统计 2026-05-25 至 2026-05-31 的官方更新，重点覆盖 Codex、ChatGPT、OpenAI API、Claude、Anthropic API、GitHub Copilot、Vercel AI Gateway、Vercel Sandbox 和 Gemini CLI 这些对 AI 编程、模型调用和 agent 开发工作流有直接影响的变化。

<!--truncate-->

## AI 编程工具

### Codex

OpenAI 在 5 月 28 日的 [ChatGPT Release Notes](https://help.openai.com/en/articles/6825453-chatgpt-release-notes) 中继续更新 Codex 相关体验，重点是语音输入、图像输入、OpenAI API 支持，以及编辑器集成的可用性。

本周比较值得关注的变化包括：

1. Codex web 可以用语音输入向 Codex 发指令。对移动办公、快速补充任务上下文、审阅长线程时追加反馈会更顺手。
2. Codex web 支持上传图片作为上下文。UI 问题、报错截图、设计稿片段、终端截图都可以更直接地交给 Codex 分析。
3. Codex web 和 IDE extension 里的 agent 现在可以使用 OpenAI API。对开发者来说，这意味着 Codex 不只是在编辑代码，也更容易围绕 API 调用、模型参数、请求调试和集成代码一起工作。
4. VS Code extension 增加截图输入，JetBrains extension 增加 IDE 诊断工具，并修复了 patch 打开、代码引用、Codex 登录等问题。

这些更新延续了 Codex 最近几周的方向：减少用户手动整理上下文的成本，让 agent 更自然地接收语音、图片、IDE 状态和 API 相关上下文。对实际开发任务来说，最直接的价值是 UI 调试、报错定位和 API 集成不再完全依赖文字描述。

### GitHub Copilot

GitHub 本周在 Copilot 方向有两类更新：模型治理和异步协作。

5 月 26 日，GitHub 发布 [target Copilot models to organizations with model rules](https://github.blog/changelog/2026-05-26-target-copilot-models-to-organizations-with-model-rules/)。企业管理员可以用 model rules 控制组织成员在 Copilot 中可用的模型范围，而不是把所有模型选择权完全交给个人用户。

这对企业团队比较重要。AI 编程工具进入团队后，模型选择会影响成本、合规、数据策略和输出稳定性。模型规则让平台团队可以按组织策略收敛模型，同时保留 Copilot 的多模型能力。

同日，Copilot coding agent 的 issue 委派流程也有更新：[assign issues to Copilot asynchronously](https://github.blog/changelog/2026-05-26-assign-issues-to-copilot-asynchronously/)。用户可以把 issue 分配给 Copilot，让它在后台分析、修改代码、开 pull request。这个工作流更接近“把明确任务交给 agent 排队处理”，而不是只在 IDE 里实时问答。

### Gemini CLI

Google 的 [Gemini CLI GitHub Releases](https://github.com/google-gemini/gemini-cli/releases) 本周继续发布多个 nightly 版本，包括 5 月 27 日的预发布版本。

这类更新不是面向所有用户的稳定大版本，但对正在试用本地命令行 AI agent 的开发者有参考价值：Gemini CLI 仍在快速补齐命令行交互、工具调用和本地开发工作流能力。生产团队不建议直接依赖 nightly；如果你在评估本地 AI coding agent，可以把它作为观察 Gemini agent 工具链演进速度的信号。

## 模型平台

### ChatGPT

ChatGPT 本周的主要开发者相关变化仍然集中在 Codex 入口上。OpenAI 在 5 月 28 日的 [ChatGPT Release Notes](https://help.openai.com/en/articles/6825453-chatgpt-release-notes) 中提到，Codex web 新增语音、图片输入，并且 Codex web 与 IDE extension 的 agent 可以使用 OpenAI API。

这说明 ChatGPT 正在从“对话入口”继续向“开发任务控制台”靠近。用户可以在 ChatGPT 相关入口里给 Codex 补充更多上下文，再让它进入代码、浏览器、IDE 和 API 调试链路。

对 AI 产品用户来说，影响不只是功能数量增加，而是上下文入口变多：文字、语音、图片、IDE 诊断、API 调用问题都可以进入同一个 agent 线程，减少跨工具复制粘贴。

### Claude

Anthropic 在 5 月 28 日的 [Claude Release Notes](https://support.claude.com/en/articles/12138966-release-notes) 中发布了 Claude Code 的 Secure Code Mode。这个模式面向 Team 和 Enterprise 计划，用来限制 Claude Code 在代码库、终端命令和外部资源访问上的权限。

对企业开发团队来说，这是一个很实际的补齐项。AI 编程 agent 越能执行命令、读取文件和改代码，安全边界就越重要。Secure Code Mode 的意义在于让管理员可以更明确地控制 Claude Code 能访问什么、能执行什么，而不是只靠使用规范约束。

如果团队正在把 Claude Code 引入公司项目，这类权限能力会直接影响能否进入真实仓库和 CI/CD 环境。个人用户感知不一定强，但对安全、平台工程和合规团队很关键。

## API/SDK

### OpenAI API

OpenAI 在同一条 [ChatGPT Release Notes](https://help.openai.com/en/articles/6825453-chatgpt-release-notes) 中提到 Codex web 和 IDE extension agent 现在可以使用 OpenAI API。虽然这不是单独的模型发布，但对开发者工作流很关键：Codex 可以更直接地围绕 API 集成任务工作，例如检查 SDK 用法、调整请求参数、定位响应错误、补齐示例代码。

这类变化对使用 OpenAI API 的团队意味着：AI 编程助手和模型平台的边界正在变薄。过去 API 文档、代码编辑器、调试日志、聊天助手经常是分开的；现在 Codex 更适合把这些上下文放到同一条任务链路里处理。

升级或迁移上没有看到额外强制动作，但团队仍然需要按自身项目的密钥管理和权限策略使用 API。尤其是在 IDE agent 场景里，应避免把生产密钥、客户数据或不必要的私有环境变量直接暴露给 agent 线程。

### Anthropic API

Anthropic 本周的官方更新主要集中在 Claude Code 的权限控制，而不是 Anthropic API 的新模型或接口变更。[Claude Release Notes](https://support.claude.com/en/articles/12138966-release-notes) 中的 Secure Code Mode 更偏产品与企业治理，但它会影响使用 Claude Code 连接企业代码库和内部工具的落地方式。

对开发平台团队来说，值得关注的不是“多了一个 API 参数”，而是 Claude 生态正在把 agent 执行权限纳入企业管理。后续如果把 Claude Code、Claude API 和内部 MCP/工具调用串起来，权限边界、审计和数据访问策略需要一起设计。

### Vercel AI Gateway

Vercel 在 5 月 27 日发布 [AI Gateway model web search capability](https://vercel.com/changelog/ai-gateway-model-web-search-capability)。AI Gateway 开始支持带 web search 能力的模型，开发者可以通过统一的 Gateway 入口调用具备联网检索能力的模型。

这对做 AI 应用的团队有两个价值。第一，检索能力不必完全自己搭搜索链路，部分模型可以直接通过 provider 能力补足实时信息。第二，AI Gateway 继续强化“统一模型入口”的定位，让团队更容易在不同模型、不同供应商能力之间切换。

需要注意的是，web search 能力通常会影响响应延迟、成本、引用质量和合规边界。生产使用前应明确哪些请求允许联网、是否需要展示来源、是否要把搜索能力限制在特定功能里。

### Vercel AI SDK

本周没有看到 Vercel AI SDK 本体的稳定大版本发布，但 AI Gateway 的 web search 能力会影响基于 Vercel AI SDK 构建的应用架构：如果应用已经通过 Vercel AI Gateway 统一转发模型请求，就可以在能力可用时把联网模型接入现有生成链路。

因此，本周对 Vercel AI SDK 用户的实际建议是先评估 Gateway 层，而不是急着改 UI 或消息协议。需要重点回归 streaming、tool calling、引用展示、缓存策略和费用监控这些位置。

## Agent 生态

### Vercel Sandbox

Vercel 在 5 月 28 日发布 [Vercel Sandbox is now available in private beta](https://vercel.com/changelog/vercel-sandbox-is-now-available-in-private-beta)。Sandbox 面向需要执行不受信任代码、运行 agent 任务或隔离构建过程的场景，当前处于 private beta。

这个方向对 AI agent 产品很重要。越来越多 coding agent、数据分析 agent、自动化修复工具都需要“运行代码”而不只是“生成代码”。一旦 agent 能执行用户提供的脚本、依赖安装或测试命令，就必须有隔离环境、资源限制和生命周期管理。

可用性方面，本周公告明确是 private beta，不是所有 Vercel 用户都能直接使用。对正在做 AI coding agent、在线 IDE、代码评测、自动化 PR 修复的团队，可以关注申请入口和后续公开可用时间。

### MCP 生态

本周在 Model Context Protocol 官方站点和核心 GitHub 仓库里没有看到 2026-05-25 至 2026-05-31 期间足够明确、适合单独展开的正式发布。不过从 Codex、Claude Code、Vercel Sandbox 这些更新可以看到同一条趋势：agent 工具调用、权限边界和隔离执行环境正在成为开发者产品的基础能力。

对开发者来说，MCP 相关选型本周不需要因为某个新版本立即迁移；更值得做的是梳理内部工具暴露给 agent 时的权限、审计和最小上下文原则。

## 本周观察

本周 AI 产品更新的主线不是单一模型能力爆发，而是“agent 进入真实开发环境”后的基础设施补课。

Codex 和 ChatGPT 在补语音、图片、IDE、API 这些上下文入口；GitHub Copilot 在补企业模型治理和异步 issue 处理；Claude Code 在补安全权限；Vercel 则把 AI Gateway 和 Sandbox 分别推向模型调用统一入口与安全执行环境。

对开发者和 AI 产品团队来说，下一阶段选型可以重点看四个问题：

1. agent 能不能拿到足够准确的上下文。
2. 模型和工具调用能不能被团队治理。
3. 代码执行有没有隔离环境和权限边界。
4. API、IDE、浏览器、终端、CI 能不能组成连续工作流。
