2026 年 7 月 8 日，TypeScript 团队正式发布 7.0 版本。这不仅是版本号的一次跃进——核心编译器从 TypeScript 自举实现完全迁移至 Go，带来了数量级的性能提升，也标志着 TypeScript 进入原生编译时代。

<!--truncate-->

## 性能飞跃：从分钟到秒

TypeScript 7.0 最大的变化在于将编译器用 Go 重写，利用原生代码执行、共享内存多线程和现代硬件优化手段，全量构建速度达到 6.0 的 8-12 倍。

| 项目 | TS 6 构建时间 | TS 7 构建时间 | 提速 |
|------|-------------|-------------|------|
| VS Code | 125.7s | 10.6s | 11.9x |
| Sentry | 139.8s | 15.7s | 8.9x |
| Bluesky | 24.3s | 2.8s | 8.7x |
| Playwright | 12.8s | 1.47s | 8.7x |
| tldraw | 11.2s | 1.46s | 7.7x |

内存方面也有显著改善：VS Code 项目内存占用从 5.2GB 降至 4.2GB（-18%），Bluesky 从 1.8GB 降至 1.3GB（-26%）。编辑器体验的提升更为直观——在 VS Code 代码库中打开一个包含错误的文件，首次报错从约 17.5 秒缩短到 1.3 秒以内，提速超过 13 倍。

### 并行度控制

7.0 将解析、类型检查和代码生成的各个阶段并行化，并提供了三个控制参数：

- **`--checkers`**：类型检查器工作线程数，默认 4。设置为 8 时 VS Code 可进一步从 10.6s 降至 7.51s。设置为 1 则退化为单线程类型检查。
- **`--builders`**：`--build` 模式下项目引用的并行构建数，与 `--checkers` 相乘。`--checkers 4 --builders 4` 允许同时运行最多 16 个类型检查器。
- **`--singleThreaded`**：完全禁用并行化，所有工作限制为单线程，用于调试或资源受限环境。

## 全新的 `--watch` 模式

文件监听器也经过彻底重写，移植自 Parcel 的 C++ 文件监听器（Go 版本附带少量汇编适配层）。Go 标准库没有内置的文件监听 API，纯轮询方案开销过大，此移植解决了这些根本性问题。

## 安装方式

```bash
npm install -D typescript
```

TS 7 提供独立的 `tsc` 可执行文件。VS Code 有专门扩展，Visual Studio 按工作区自动启用。夜间构建可通过 `typescript@next` 标签获取。

## 破坏性变更

### 来自 6.0 的弃用项现在变为硬错误

以下选项在 6.0 中已弃用，7.0 中完全移除：

- `target: es5` 不再支持
- `downlevelIteration` 不再支持
- `moduleResolution: node` / `node10` / `classic` 不再支持，应使用 `nodenext` 或 `bundler`
- `module: amd` / `umd` / `systemjs` / `none` 不再支持，应使用 `esnext` 或 `preserve`
- `baseUrl` 不再支持
- `esModuleInterop` 和 `allowSyntheticDefaultImports` 不能设为 `false`
- `alwaysStrict` 默认为 true 且不能关闭
- `import` 断言（`asserts` 关键字）不再支持，应使用 `with` 语法

### 默认配置变化

- `strict` 默认启用
- `module` 默认为 `esnext`，`target` 默认为当前稳定的 ECMAScript 版本
- `noUncheckedSideEffectImports` 默认为 true
- `stableTypeOrdering` 默认为 true 且无法关闭
- **`rootDir` 默认变为 `./`**：之前自动推断源目录，现在需要显式设置（例如 `"./src"`）
- **`types` 默认变为 `[]`**：以前自动加载所有 `@types` 包，现在需要手动列出

这两项默认值变动是迁移中最容易出问题的地方——缺失 `@types` 或未正确设置 `rootDir` 都会导致编译失败。

### 模板字面量类型的 Unicode 处理

```typescript
type HeadTail<S> = S extends `${infer Head}${infer Tail}` ? [Head, Tail] : never;
type Result = HeadTail<"😀abc">;
// TS 7：["😀", "abc"]
// TS 6：["\ud83d", "\ude00abc"]
```

7.0 将 emoji 等 Unicode 码点视为一个整体处理，与 `for...of` 和扩展运算符的行为一致。这对构建在模板字面量类型上的类型体操有一定影响。

### JavaScript / JSDoc 支持调整

JS 文件的分析现在与 `.ts` 文件更加一致。一些不再支持的模式：

- 不能将值用作类型（需要使用 `typeof`）
- `@enum` 不再被特殊识别
- 单独的 `?` 不再作为类型使用
- `@class` 不会将函数标记为构造函数
- 闭包风格的函数语法 `function(string): void` 不再支持

## 内嵌语言项目的现状（Vue / Svelte / Angular / Astro）

**一个重要的限制**：TypeScript 7.0 不包含稳定的程序化 API——该功能计划在 7.1 中加入。这意味着依赖 TypeScript API 的工具（如 Volar）目前只能继续使用 6.0。

- **Angular**：可在 CLI 中使用 TS 7 进行类型检查，编辑器内仍使用 TS 6
- **Vue / MDX / Astro / Svelte**：暂时继续使用 6.0
- VS Code 用户可通过 "Disable TypeScript 7 Language Server" 退回到 6.0

### 并排运行方案

微软发布了 `@typescript/typescript6` 包，提供 `tsc6` 可执行文件并重新导出 6.0 API。可通过 npm alias 实现并排安装：

```json
{
  "devDependencies": {
    "@typescript/native": "npm:typescript@^7.0.2",
    "typescript": "npm:@typescript/typescript6@^6.0.2"
  }
}
```

这样一来脚本和构建工具可以 `import` 6.0 API，命令行则使用 7.0。

## 生态兼容性

TS 7 经过数万项测试以及面向 GitHub 上主流 TypeScript/JavaScript 代码库的模糊测试。实际用户反馈包括：

- **Slack**：消除了 40% 的合并队列等待时间，CI 类型检查从 7.5 分钟降至 1.25 分钟
- **Vanta**：最大项目之一提速 9 倍
- **Canva**：编辑器首次报错从 58 秒降至 4.8 秒
- **Microsoft News Services**：每月节省 400 小时的 CI 等待时间
- **PowerBI 工程师**：将编辑器中的体验描述为"救命级"改进

## 后续规划

- **7.1**：预计发布新的程序化 API，打通生态工具（typescript-eslint、Volar 等）的接入路径
- **发布节奏**：保持每 3-4 个月一个 feature 版本，与 7.0 之前的节奏一致
- **方向**：新特性、易用性改进、更多性能优化

## 迁移建议

1. **先升级到 6.0**：处理所有弃用警告，确保 `stableTypeOrdering` 开启，清理 `ignoreDeprecations` 配置
2. **检查 `tsconfig.json`**：显式设置 `rootDir`（如 `"./src"`）和 `types` 列表
3. **安装 TS 7**：`npm install -D typescript`，运行 `npx tsc --showConfig` 确认配置生效
4. **对使用内嵌语言的项目**：保持 6.0 作为编辑器工具链，仅在 CI/CLI 中使用 7.0
